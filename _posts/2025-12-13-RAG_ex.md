# Vector DB 없이 RAG를 구성한 사례

최근 **한 배달 플랫폼 업체의 기술블로그**에서  
Vector DB 없이 RAG(Retrieval-Augmented Generation)를 구성한 사례를 발견했다.  
👉 https://techblog.woowahan.com/23836/

일반적으로 RAG는 임베딩과 Vector DB를 활용한 검색 구조를 떠올리지만,  
이 사례는 **RAG의 본질에 집중한 매우 단순한 접근**을 선택했다.

---

## 핵심 아이디어

- UX 라이터가 직접 관리하는 가이드·정책·톤 규칙을
- 문서 저장소가 아닌 **프롬프트용 지식**으로 정리
- 필요한 지식을 **검색 결과처럼 프롬프트에 보강**
- LLM은 해당 컨텍스트를 기반으로 응답만 생성

즉,  
**검색(Retrieval)을 시스템이 아니라 사람이 수행**한 RAG 구조다.

---

## 구조 요약 (LangGraph 관점)

```

사용자 질문
↓
질문 유형 분류
↓
유형별 컨텍스트 보강
↓
LLM 응답 생성

```

- 질문 유형 분류: 정책 / 가이드 / 오류 / 가격 등
- 컨텍스트 보강: 사람이 정의한 규칙 기반 지식 선택
- 답변 생성: LLM은 작성 역할만 수행

---

## 왜 이게 RAG인가?

- 모델 내부 지식이 아닌 **외부 지식(가이드)**를
- 응답 시점에 **컨텍스트로 주입(Augmentation)**하고
- 그 기반으로 **생성(Generation)**하기 때문

Vector DB는 없지만,  
RAG의 개념적 정의에는 정확히 부합한다.

---

## 배운 점

- RAG는 기술 스택이 아니라 **사고 방식**
- 작은 규모·정확성이 중요한 영역에서는
  사람 보강 RAG가 더 효과적일 수 있음
- 필요해질 때 Vector DB를 **확장 요소로 추가**하면 충분

> RAG가 어렵게 느껴진다면,  
> 이렇게 단순한 구조로 시작하는 것도 충분히 좋은 출발이다.
